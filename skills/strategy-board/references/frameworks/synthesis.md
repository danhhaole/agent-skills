# Synthesis & Communication — Minto's Working File

Tools for turning the engagement's analysis into documents executives can
act on. The discipline: think top-down, write answer-first, keep every number
traceable to `fact-base.md`.

## The Pyramid

**When:** structuring `recommendation.md`, a deck, or any argument longer
than a paragraph.

**How:** one **governing thought** at the top — the recommendation as a full
sentence with the "because" in it ("Enter Vietnam via acquisition, because
organic entry cannot reach viable scale before the incumbent closes the
channel"). Below it, the **key line**: 3–4 supporting arguments that are
MECE — no overlaps, no gaps — each a claim, not a topic ("The market
window closes in ~2 years", not "Market timing"). Below each claim, its
evidence `[S#]`. Test bottom-up: does each group of evidence *force* its
claim? Do the claims together *force* the governing thought? Where they
don't, the gap is an analysis gap, not a writing gap — send it back to the
relevant specialist rather than papering over it with prose.

## Issue tree (problem structuring)

**When:** Phase 0–1, when the question is broad or the engagement keeps
drifting — the tool that turns "what should we do about X?" into researchable
pieces. Drucker frames the question; Minto owns the decomposition.

**How:** put the decision question at the root, break it into 2–4 MECE
sub-questions (no overlaps, no gaps — test: would answering all of them
force the answer to the root?), and recurse one or two levels until each
leaf is either researchable (goes to the fact base) or judgment the
executive must supply. Hypothesis-first variant: state the candidate answer,
then build the tree of what must be true (which hands over to Wack's
WWHTBT).

**Output format:** the indented tree, each leaf tagged `[research → S#]`,
`[assumption → A#]`, or `[executive judgment]`; then one line naming which
branch is load-bearing and gets analyzed first (80/20 — not every branch
deserves equal depth).

## SCQA opening

**When:** the first four sentences of every deliverable.

**How:** **Situation** — where we are, stated so the reader nods.
**Complication** — what changed or threatens, creating urgency.
**Question** — the decision this raises (should match `brief.md`; if it
doesn't, the engagement drifted — flag it). **Answer** — the governing
thought. After SCQA, the reader knows the ask; everything else is support.

## Options table (Phase 3 → Phase 5 carry-over)

**When:** presenting the choice honestly — in `options.md` and again,
compressed, in the recommendation.

**Output format:**

```
| | Option A | Option B | Option C |
|---|---|---|---|
| In one sentence | | | |
| Investment (range) | | | |
| Return profile / payback | | | |
| Downside case | | | |
| Must be true | | | |
| Capability demand | | | |
| Reversibility | | | |
| Main risk | | | |
```

Same rows for every option, rejected options included in the final document
with one honest line each on why not — a board that can see what was
rejected trusts what was chosen.

## One-pager

**When:** the executive needs the whole engagement on one page.

**Format:** SCQA (4 sentences) → the recommendation in bold → three key-line
claims with one supporting number each → what we will NOT do → risks
accepted (from the pre-mortem, 2–3 lines) → the decision requested and by
when. Nothing else. If it doesn't fit, the pyramid isn't finished.

## Deck storyline

**When:** a board presentation is the delivery vehicle.

**How:** write all slide *titles* first, as claims — read in sequence, the
titles alone must carry the entire argument (the "elevator test": someone
reading only titles reaches the same conclusion). One idea per slide; the
body of each slide exists only to prove its title. Typical spine: SCQA →
situation evidence → complication evidence → options honestly compared →
recommendation → what must be true & risks accepted → roadmap & ask.

## SWOT on request

Executives ask for SWOT by name; the board doesn't refuse — it upgrades it.
Never brainstorm a SWOT from a blank page: build it as a *synthesis view*
over existing artifacts — strengths/weaknesses distilled from the capability
assessment and performance diagnosis, opportunities/threats from the market
and scenario work — every cell tracing to `[S#]`/`[A#]`, 3–4 items per
quadrant, each phrased as a claim with a consequence ("Cost position 15%
below peers [S3] → can absorb a price war that breaks smaller rivals"), and
closed with a So-what. A SWOT that merely lists is a shopping bag, not
analysis.

## One document for the executive

However rich the engagement folder, the executive's deliverable is **one
self-contained document** — `recommendation.md` for a full engagement, a
single memo for a single-analysis engagement (finding, evidence, So-what,
recommended action — with the working files linked as appendices, never
required reading). A busy board member should be able to act from the memo
alone on a flight; the folder exists for the one who asks "show me". When a
deck or visual dashboard is the vehicle and rendering skills are available
in the environment (e.g. a slide or dataviz skill), Minto writes the
storyline and the sourced numbers, then hands rendering to that skill — she
owns the argument, not the pixels.

**Tips:** kill every section that doesn't change the decision (the So-what
test, applied with a knife). Keep Taleb's surviving objections in the
document — pre-answered objections build more confidence at the board table
than polished certainty. And numbers in the final document keep their
`[S#]` marks; "trust me" is not a citation.
